# Input Contract

This harness is file-based. A case is reproducible only when the inputs,
configuration, generated artifacts, tool versions, and human decisions are
recorded under the case directory.

## Required Inputs

Every executable case needs:

- Case config: YAML file derived from `configs/default.yml`.
- Design request: plain text in `inputs.request`.
- Stackup: YAML file with layer order, layer type, thickness, material, and
  reference-plane assumptions.
- Port intents: JSON file defining solver port names, signal/reference nets,
  launch locations, reference geometry selection, impedance, role, and expected
  port order.

For a routed PCB/package case, one of the following must also exist:

- KiCad board/project generated by the harness.
- Imported vendor database such as ODB++ or IPC-2581.
- Pre-existing AEDB/AEDT project with verified ports.
- Touchstone file when running only the circuit/spec-check stage.

## Optional Inputs

- Governing specification PDF.
- User-approved reference documents.
- Web research summaries.
- Pre-authored `design_strategy.yaml`.
- Bump map, pin map, connector pinout, or lane map CSV/XLSX.
- ADS workspace template or existing verification schematic.
- Waiver file for engineering exceptions.

Large PDFs, books, proprietary board databases, and generated solver outputs
must stay outside Git unless redistribution rights are explicit.

## Path Rules

- Relative paths are resolved from the config file directory.
- Absolute paths are allowed for local tool inputs.
- `<repo>/...` may be used in documentation to indicate a repository-relative
  path, but scripts should receive real resolved paths.
- Generated outputs go under `outputs/<case_name>/` unless the config overrides
  `harness.output_root`.

## Units

- Geometry defaults to millimeters.
- EDB/PyAEDT APIs often require meters internally; adapters must convert units
  explicitly.
- Frequency defaults to GHz.
- Data rate is in GT/s or Gbps as stated by the governing spec.
- Time-domain eye width must be reported in UI and the time unit used by the
  simulator.

## Port Intent JSON

Required fields per port:

- `name`
- `type`: `edb_polygon_edge` by default
- `net`
- `reference_net`
- `positive_layer`
- `negative_layer`
- `positive_x`
- `positive_y`
- `impedance_ohm`
- `role`: for example `tx_near`, `rx_far`, `aggressor`, `victim`
- `expected_order`

Recommended fields per default HFSS 3D Layout polygon-edge port:

- `positive_x`, `positive_y`: signal launch coordinate used to select the
  nearest signal polygon/primitive edge.
- `reference_net`, `negative_layer`: local reference geometry source used to
  select the nearest reference polygon/primitive edge.
- `placement_rule`: `polygon_edge_signal_to_local_reference_edge`.
- `port_method`: `edb_polygon_edge` unless the case explicitly overrides it.
- `reference_selector`: optional structured hint such as `nearest_solid_edge`,
  `same_x_projection`, `same_y_projection`, or a converter-specific primitive
  and edge id. Prefer this over coordinate-port reference terminals.

Coordinate-port override fields, for manual/debug use only:

Coordinate-port override is disabled in normal harness runs. It requires
`--allow-coordinate-port-override`, a written reason in the case report, and a
verified non-empty Touchstone before ADS handoff.

Default polygon-edge example:

```json
{
  "name": "DATA0_P1",
  "type": "edb_polygon_edge",
  "net": "DATA0",
  "reference_net": "GND",
  "positive_layer": "F.Cu",
  "negative_layer": "In1.Cu",
  "positive_x": 2.000,
  "positive_y": 2.540,
  "impedance_ohm": 50,
  "role": "tx_near",
  "expected_order": 1,
  "placement_rule": "polygon_edge_signal_to_local_reference_edge",
  "reference_selector": "nearest_local_solid_reference_edge"
}
```

This is the normal form. It intentionally has no `negative_x` or `negative_y`.
The HFSS importer selects a local signal launch edge and a nearby solid
reference primitive/edge.

- `positive_x`, `positive_y`: signal terminal coordinate on pad or trace.
- `negative_x`, `negative_y`: reference terminal coordinate on the negative
  layer. The coordinate must be on nearby reference-net copper facing the
  signal launch edge, not empty board space, and must avoid via
  holes/antipads. If the closest facing reference point is blocked by a via
  hole or antipad, move the terminal to the nearest adjacent solid reference
  copper while keeping the circuit-port span local.
- `placement_rule`: for example
  `two_point_circuit_port_ref_terminal_on_solid_plane`.

Bad pattern:

```json
{
  "positive_x": 2.03,
  "positive_y": 2.54,
  "negative_x": 2.03,
  "negative_y": 2.79,
  "placement_rule": "somewhere_on_gnd_plane"
}
```

This can be on GND copper but too far from the launch edge, producing a visible
HFSS port that solves but cannot export network data.

Coordinate override debug pattern:

```json
{
  "positive_x": 2.000,
  "positive_y": 2.540,
  "negative_x": 1.895,
  "negative_y": 2.540,
  "placement_rule": "local_two_point_circuit_port_reference_terminal_on_solid_gnd"
}
```

Use this only when `--allow-coordinate-port-override` is explicitly selected.
The negative point is local, faces the signal launch edge, and is outside via
drill/antipad keepout. If the nearest facing point is blocked, slide it along
the adjacent solid reference copper and record the clearance evidence. This is
not the preferred KiCad/AEDB handoff path.

Legacy `x`/`y` ports may be readable by older adapters, but new HFSS handoff
cases must emit polygon-edge port intents and must pass `check:port-launch`.
New cases must not emit `negative_x`/`negative_y` unless they are intentionally
using the coordinate-port override path.

Recommended top-level fields:

- `schema`: `sipi-harness.port-intents.v1`
- `unit`: `mm`
- `expected_port_count`

## Bump/Pin Map CSV

Required columns for a simple map:

- `interface`
- `side`
- `pin`
- `row`
- `column`
- `x_mm`
- `y_mm`
- `net`
- `role`
- `layer`

If a map comes from a PDF figure, the case must also record the figure/table
source, extraction method, raw evidence, rendered image/crop when available,
normalization assumptions, and reviewer status.

## Stackup YAML

Each layer entry should include:

- `name`
- `type`: `signal`, `plane`, `dielectric`, `mask`, or `assembly`
- `thickness_mm`
- `material`

Signal layers should include `reference`. Plane layers should include `net`.
Dielectrics should include `dk` and `df` when available.

## Common Input Errors

- Missing reference layer for a high-speed signal port.
- Coordinates in mil or meter but declared as mm.
- Port names not matching expected Touchstone port order.
- Bump map extracted from a figure but used without visual review.
- 4-layer board treated as four routing layers when two layers are reserved as
  planes.
- Spec PDF copied into Git without redistribution rights.
- Existing Touchstone has the wrong port count or no frequency data lines.

The harness should fail early with actionable errors for schema problems and
mark downstream stages as blocked, not silently generate proxy compliance.
